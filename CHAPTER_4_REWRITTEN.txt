CHAPTER FOUR
RESULTS AND DISCUSSION

4.1 Introduction

This chapter presents the results obtained from the implementation and evaluation of the hyper-heuristic university timetable generation system described in Chapter Three. The chapter is structured as follows. Section 4.2 provides a brief overview of the system's user interface to contextualise the results. Section 4.3 presents the testing outcomes from unit, integration, and NLP parsing tests. Section 4.4 restates the experimental setup. Sections 4.5 to 4.7 present the quantitative results from all three algorithms — the Genetic Algorithm (GA), Simulated Annealing (SA), and Tabu Search (TS) — along with comparative analysis, convergence analysis, and a sample timetable output. Section 4.8 provides the full discussion, linking every major finding back to the objectives stated in Chapter One and the literature reviewed in Chapter Two. Section 4.9 summarises the chapter.

All results presented in this chapter were obtained from actual system execution using the Covenant University academic dataset described in Section 3.3 of Chapter Three. As noted in that chapter, the dataset is partially estimated: where official records were unavailable, reasonable estimates were supplied based on the researcher's institutional knowledge as a registered student of Covenant University. The results are therefore best understood as a demonstration of the system's capabilities and algorithm behaviour on a realistic institutional dataset, rather than as a definitive audit of Covenant University's scheduling data.

The Time4Tables system was deployed locally as a two-process web application. The Streamlit frontend was launched via the command streamlit run app_ui.py, making the interactive interface accessible at http://localhost:8501. The FastAPI backend was launched concurrently via uvicorn app:app --reload, providing the RESTful scheduling API at http://127.0.0.1:8000 with auto-generated documentation at /docs.

4.2.1 Welcome and Login Interface

The welcome page displays the system branding ("Time4Tables") alongside a login panel with three access portals: Academic Administrator (username/password authentication with full system access), Lecturer (dropdown selection for personalised schedule viewing), and Student (level and department selection for filtered timetable access). The interface features a custom purple-themed design with glassmorphism card effects and animated gradient backgrounds.

4.2.2 Four-Step Administrator Workflow

After administrator login, the interface presents a four-step sequential workflow:

Step 1 — Dataset Upload: The administrator selects and uploads a CSV file containing the academic resource data. On successful upload, the system displays the dataset preview (first five rows), the total number of records loaded, and the count of unique venues and lecturers detected.

Step 2 — Hard Constraint Entry: A text area accepts natural language mandatory scheduling rules. A guidance note explains the concept of hard constraints and provides examples. On submission, the system displays a parsed constraint summary confirming what the NLP module has extracted from the input text.

Step 3 — Soft Constraint Entry: A separate text area accepts preferred scheduling preferences. The parsed soft constraints are merged with the hard constraints and displayed for confirmation.

Step 4 — Optimisation Execution: The administrator clicks "Execute Multi-Algorithm Generation." A real-time progress indicator tracks the execution of all three algorithms. On completion, the benchmark comparison table and the optimised timetable grid are displayed.

4.2.3 Timetable Visualisation

The optimised timetable is presented in a weekly grid explorer with tabbed navigation for each day (Monday through Saturday). Each tab displays a filterable, sortable table showing Course Code, Lecturer, Timeslot, Venue, Students Registered, and Venue Capacity. The administrator dashboard additionally includes the Algorithm Benchmark Table, a full Master Course Schedule, a Manual Adjustment panel, and a CSV Export button.

4.3 Testing Results

4.3.1 Unit Testing

Individual modules were tested in isolation before integration. The following units were tested:

NLP Parser (nlp_parser.py):
A series of representative English constraint sentences were passed to parse_constraints() and the output dictionary was verified against expected values. The parser correctly identified all tested constraint types with no mismatch between expected and actual output.

Fitness Function (engine.evaluate_timetable()):
Timetable configurations with predetermined clash counts were programmatically constructed and evaluated. Timetables with exactly two lecturer clashes returned a fitness contribution of 200,000 (2 × 100,000). Timetables with one unscheduled session returned a fitness contribution of 500,000. All evaluated values matched the expected calculations.

Room Selection (engine.get_valid_rooms()):
Computer science courses (prefixed CSC, CIT, CIS) were confirmed to receive COMPUTER_LAB as the top-priority venue candidate. Courses with 450 enrolled students were confirmed to exclude rooms with capacity below 450. The 70% relaxed fallback was triggered correctly when no full-capacity room was available.

Initial Solution Generator:
Generating an initial solution from the cleaned dataset produced a timetable with all 490+ courses assigned, with a small number of unscheduled sessions in early tests (before venue capacity configuration was completed). After full preprocessing and venue mapping, the unscheduled count reduced to two, which were subsequently eliminated by the optimisation algorithms.

4.3.2 Integration Testing

End-to-end workflow testing was conducted to verify that the full pipeline from user input to timetable output functions correctly:

Test Path: Administrator uploads CSV → Hard constraint text entered → Soft constraint text entered → NLP parser converts text to JSON → Initial timetable generated → All three algorithms execute → Benchmark table displayed → CSV downloaded.

Each stage was verified to pass data correctly to the next. The FastAPI endpoint at /generate was tested using both the Streamlit interface and the /docs auto-documentation interface. Request and response formats were verified for the /upload, /constraints, /generate, and /timetable endpoints.

4.5.7 Convergence Analysis

The convergence behaviour of the three algorithms reveals distinct search patterns corresponding to their underlying mechanisms.

Genetic Algorithm: The GA exhibits rapid initial improvement due to population diversity and the crossover operator combining good scheduling patterns across individuals. Fitness drops sharply in the first 20–30 generations as the population eliminates most hard constraint violations. The rate of improvement slows after generation 50, and the algorithm converges to its best solution near generation 65.

Simulated Annealing: SA shows a more gradual convergence pattern with occasional fitness increases during early high-temperature iterations, where the Metropolis criterion accepts deteriorating moves to escape local optima. As the temperature decreases through the geometric cooling schedule (T × 0.995 per iteration), the trajectory stabilises and the algorithm exploits the best region found.

Tabu Search: TS demonstrates consistent incremental improvement with occasional plateaus where all 20 generated neighbours are either tabu or inferior. The tabu list (size 30) effectively prevents revisiting, ensuring continuous exploration of new solution regions. The algorithm reached its best solution near iteration 82.

4.8 Discussion

4.8.1 Effectiveness of the Hyper-Heuristic Approach

The hyper-heuristic framework proved effective in this study. By running all three algorithms on the same problem instance and automatically selecting the best-performing solution, the system guarantees that the administrator receives the best available timetable regardless of which algorithm happens to perform best for their particular dataset and constraint configuration. This directly addresses a fundamental limitation of single-algorithm systems: the uncertainty about which algorithm will be most effective for a given scheduling instance (Burke et al., 2013).

The results confirm this value empirically. The three algorithms produced distinctly different final fitness scores (GA: 150,000; TS: 250,000; SA: 350,000), a 133% performance spread between the best and worst results. If only SA had been available, the administrator would have received a timetable with one lecturer clash and one room clash still unresolved. The hyper-heuristic selection mechanism ensured that the GA's superior result was delivered instead, with all critical hard constraint violations eliminated.

4.8.2 Why the Genetic Algorithm Performed Best

The GA's superiority over SA and TS for this problem instance can be attributed to four structural advantages:

Population Diversity: GA maintains 50 distinct timetable solutions simultaneously, enabling the search to explore multiple regions of the solution space in parallel. SA and TS maintain only a single current solution, fundamentally limiting their breadth of exploration.

Crossover Recombination: The single-point crossover operator creates children that inherit good scheduling patterns from two different parent solutions. When Parent 1 has efficient Monday/Tuesday assignments and Parent 2 has efficient Thursday/Friday assignments, their offspring may combine both. SA and TS have no equivalent recombination mechanism.

Elitism: The top five solutions are preserved intact across every generation, ensuring that the best fitness achieved is monotonically non-increasing. Once the GA eliminates all lecturer clashes (around generation 40), it cannot regress to a state with lecturer clashes, because at least one clash-free solution is always preserved in the elite.

Balanced Mutation: The 20% mutation rate provides effective exploration (modifying 20% of sessions each generation) while retaining 80% of each solution's structure. This balance prevents both stagnation (too little mutation) and random destruction (too much mutation).

These structural advantages are consistent with findings in the literature. Burke et al. (2013) noted that population-based methods tend to outperform single-solution methods on timetabling problems because of the richer information exchange enabled by a diverse population.

4.8.3 Why Simulated Annealing Underperformed Relative to GA

SA's final fitness of 350,000 (86.27% improvement) is significantly behind GA's 94.12% improvement. Several factors explain this gap:

Single-Solution Trajectory: SA has no mechanism for simultaneously exploring multiple solution regions. Its search is confined to a single path through the solution space, making it vulnerable to the particular local optima encountered during any given run.

Acceptance of Deteriorating Moves: While the Metropolis acceptance criterion is designed to escape local optima by occasionally accepting worse solutions, it can also lead the search away from a promising solution region when temperature remains high, as demonstrated by occasional fitness spikes in the convergence profile.

Parameter Sensitivity: SA's performance is sensitive to the cooling schedule. The chosen parameters (T0 = 10,000, α = 0.995, 500 iterations) may not have been optimal for this specific dataset size and constraint configuration. Adaptive cooling schedules, as described by Kiefer, Hartl, and Schnell (2017), might improve SA's performance in future work.

Despite these limitations, SA's 30-second execution time makes it attractive in contexts where speed is prioritised over solution quality, such as rapid re-scheduling in response to last-minute changes.

4.8.4 Tabu Search Performance Assessment

TS achieved intermediate performance (fitness 250,000, 90.20% improvement), outperforming SA but remaining below GA. Its effective cycling prevention (tabu list size 30) ensured continuous exploration and avoided revisiting recently visited solutions, consistent with Sörensen et al. (2018)'s description of TS's memory-based search advantages. However, TS was constrained by its neighbourhood structure: generating only 20 neighbours per iteration limits the diversity of each search step. Compared to GA's population of 50, TS effectively has a smaller "view" of the solution space per iteration. Increasing the neighbours per iteration (at the cost of longer execution time) or using adaptive tabu tenure (as studied by Lü and Hao in benchmark timetabling) could improve TS's performance.

TS also required the longest execution time (~55 seconds) because each of its 300 iterations generates 20 neighbours and computes timetable signatures for all of them. For institutions where execution time is not a critical constraint, TS may still be preferred over SA due to its more consistent improvement trajectory.

4.8.5 Impact of NLP Integration on Usability

The integration of natural language processing for constraint specification is the feature that most directly addresses the accessibility barrier identified in the problem statement. The regex-based NLP parser successfully translated all six tested constraint patterns from plain English into machine-readable constraint dictionaries with 100% accuracy within the supported vocabulary. This eliminates the need for administrators to learn any formal constraint language or configuration file format.

Compared to transformer-based approaches (Pallagani et al., 2024; Abgaryan et al., 2024), the rule-based implementation offers deterministic, reproducible results with no dependency on external API calls or pre-trained model weights. As noted by Nguyen et al. (2022), rule-based NLP remains the preferred approach for structured constraint formalisation where the vocabulary is well-defined and accuracy must be guaranteed.

The key limitation of the current implementation is its fixed vocabulary: constraint types not covered by the predefined regex patterns are silently ignored. Extending the vocabulary to cover additional constraint types (for example, consecutive-hour blocking or department-specific preferences) requires adding new regex pattern groups to nlp_parser.py.

4.8.6 Capacity Violations as the Most Persistent Challenge

All three algorithms retained exactly three capacity violations at their termination, despite substantially different search strategies. This convergence on the same outcome is informative: it suggests that the remaining three capacity violations are not algorithmic failures but structural constraints of the dataset. When all appropriately sized venues for a very large course (for example, 430+ students) are simultaneously occupied in a given timeslot, no algorithm can eliminate the capacity violation without violating a harder constraint (such as splitting the class into two sections, which is outside the current system's scope). This finding aligns with the dataset limitation acknowledged in Section 3.3.1: the partially estimated venue capacity figures mean some room capacities may be slightly underestimated, which would artificially inflate the capacity violation count.

4.8.7 Scalability Considerations

The system demonstrated adequate performance for the Covenant University dataset (490+ courses, 100+ lecturers, 50+ venues) on standard consumer hardware. GA's 200 generations × 50 evaluations = 10,000 total fitness function calls completed in approximately 45 seconds. For larger institutions with 1,000+ courses, the computational cost would scale roughly linearly with dataset size (since each fitness evaluation iterates through all sessions). Potential optimisation strategies for larger deployments include parallel fitness evaluation across population members (using Python's multiprocessing library), incremental fitness updates that recalculate only affected sessions after mutation, and adaptive population sizing that reduces population when convergence is detected.

4.8.8 Comparison with Related Work

The results obtained in this study are consistent with findings in the existing literature:

Burke et al. (2013) demonstrated that hyper-heuristic portfolio approaches outperform single-algorithm systems on timetabling problems by adapting to problem-specific characteristics — a finding confirmed by the 133% performance spread between GA and SA in this study.

Ceschia et al. (2023) established the NP-hard nature of the university course timetabling problem, justifying the use of meta-heuristic approaches. The inability of any algorithm to find a perfect zero-violation solution in this study (three persistent capacity violations remained) is consistent with the theoretical expectation that finding a globally optimal solution is computationally intractable for real-world problem instances.

Katoch et al. (2021)'s review of genetic algorithms documented the superior solution quality achieved through crossover and elitism, consistent with GA's outperformance of both SA and TS in this study.

Delahaye et al. (2019) demonstrated SA's effectiveness for combinatorial optimisation while acknowledging its parameter sensitivity — consistent with SA's underperformance relative to GA under the chosen cooling parameters.

Sörensen et al. (2018) documented TS's effective cycling prevention through tabu lists, which is confirmed by TS's consistent incremental improvement trajectory and its outperformance of SA in this study.

Nguyen et al. (2022)'s demonstration of rule-based NLP accuracy for structured constraint vocabularies is confirmed by the 100% accuracy observed in the NLP parsing tests.

4.9 Summary

This chapter presented the implementation and testing details, experimental setup, quantitative results, and comprehensive discussion for the hyper-heuristic university timetable generation system. The Genetic Algorithm achieved the best overall performance, with a fitness score of 150,000 — representing a 94.12% improvement over the greedy initial solution — and complete elimination of all lecturer clashes and room clashes. Simulated Annealing achieved an 86.27% improvement, while Tabu Search achieved a 90.20% improvement. All three algorithms successfully scheduled all 490+ course sessions with zero unscheduled sessions, confirming that each algorithm is capable of producing a complete, usable timetable. The NLP constraint parser demonstrated 100% accuracy across all tested constraint patterns. Three capacity violations persisted across all algorithms, reflecting a structural constraint of the partially estimated dataset rather than algorithmic failure. The hyper-heuristic selection mechanism ensured that the best-available solution (from GA) was automatically delivered to the administrator. These results address all four research objectives established in Chapter One and are consistent with the theoretical and empirical findings reported in the literature reviewed in Chapter Two. Chapter Five will summarise the key contributions of the study, state its limitations, and present recommendations for future work.
