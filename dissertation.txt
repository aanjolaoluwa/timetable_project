CHAPTER THREE
METHODOLOGY
3.1 Introduction
This chapter gives a comprehensive description of the methodology adopted in the design, development, and implementation of the hyper-heuristic university timetable generation system with the project topic titled "From Prompt to Timetable: Integrating Natural Language Reasoning with Different HyperHeuristic Algorithms." The chapter details the research design, system analysis, proposed system architecture, algorithm design, constraint modelling, fitness function formulation, dataset description, and implementation procedures. All diagrams, pseudocode, and mathematical formulations presented in this chapter are derived directly from the actual implementation of the system.
The methodology follows a systematic software engineering approach integrating Natural Language Processing (NLP) for constraint specification with three distinct meta-heuristic optimization algorithms namely Genetic Algorithm (GA), Simulated Annealing (SA), and Tabu Search (TS), to generate conflict-free university timetables. This approach aligns with the hyper-heuristic paradigm, which selects or generates heuristics to solve optimization problems rather than directly solving the problem with a single heuristic (Burke et al., 2013).
3.2 Research Design
The research design adopted for this study is the Design Science Research (DSR) methodology. According to Dresch et al. (2015), DSR is appropriate for studies that aim to design and evaluate IT artifacts intended to solve identified organizational problems. This methodology is particularly suitable for this project because:
Problem Identification: The university timetable scheduling problem (UTSP) is a welldocumented NP-hard combinatorial optimization problem (Ceschia et al., 2023) requiring intelligent algorithmic solutions.
Artifact Design: The system constitutes a novel software artifact that integrates NLP-based constraint parsing with multiple hyper-heuristic algorithms.
Evaluation: The system's effectiveness is measured through quantitative fitness metrics, clash reduction rates, and comparative algorithm performance analysis.
The research follows an experimental approach, wherein three optimization algorithms are systematically applied to the same dataset and their performance compared using standardized evaluation metrics including fitness scores, lecturer clash counts, room clash counts, and capacity violation counts.
3.3 System Analysis
3.3.1 Analysis of the Existing System
Traditional university timetable construction at most Nigerian institutions, including Covenant University (the institution whose academic dataset forms the basis of this study), is predominantly manual or semi-automated. The existing system relies on spreadsheet-based methods where academic administrators manually assign courses to timeslots and venues. This approach suffers from several critical limitations:
High Error Rate: Manual scheduling is prone to lecturer clashes (a lecturer assigned to two courses at the same time), room conflicts (two courses assigned to the same room at the same time), and venue capacity violations (students exceeding room capacity).
Time Intensive: The process typically requires several weeks of iterative adjustments before producing a feasible timetable.
Scalability Issues: As the number of courses, lecturers, venues, and student enrolments increases, the combinatorial complexity grows exponentially, making manual approaches impractical.
Lack of Optimization: The existing system produces feasible timetables at best but does not optimize for quality metrics such as minimizing room changes, distributing classes evenly across the week, or respecting soft preferences.
No Natural Language Interface: Administrators must translate institutional policies and scheduling preferences into rigid spreadsheet formats, with no mechanism for expressing constraints in natural language.
3.3.2 Analysis of the Proposed System
The proposed system addresses the identified limitations through an intelligent, automated timetable generation platform with the following capabilities:
Natural Language Constraint Specification: Administrators input scheduling constraints in plain English (e.g., "No lecturer should teach two classes at the same time," "Morning classes are preferred"), which are parsed by the NLP module and translated into machinereadable constraint objects.
Multi-Algorithm Optimization: Three hyper-heuristic algorithms (GA, SA, and TS) are executed on the scheduling problem, and the best-performing solution is automatically selected.
Automated Clash Detection: The fitness function systematically identifies and penalizes lecturer clashes, room clashes, capacity violations, and NLP constraint violations.
Web-Based Interface: A Streamlit-powered user interface provides role-based access for administrators, lecturers, and students with real-time timetable visualization.
RESTful API Backend: A FastAPI backend enables programmatic access to the scheduling engine, supporting dataset upload, constraint parsing, timetable generation, and retrieval.
3.4 Proposed System
3.4.1 System Overview
The proposed system, named "OptimaSched" (also referred to as "Time4Tables" in the user interface), is a multi-layered intelligent timetable scheduling platform. The system accepts academic resource datasets in CSV format, parses natural language constraints from administrators, generates an initial feasible timetable, and then applies three competing optimization algorithms to produce an optimized, clash-minimized schedule.
The system operates through a four-step workflow:
Step 1 — Dataset Upload: The administrator uploads a CSV file containing course codes, lecturers, student counts, venue assignments, and course load information.
Step 2 — Hard Constraint Definition: The administrator specifies mandatory institutional rules in natural language (e.g., "No overlapping classes for lecturers").
Step 3 — Soft Constraint Definition: The administrator specifies preferred but nonmandatory scheduling preferences (e.g., "Morning classes are preferred").
Step 4 — Multi-Algorithm Optimization: The system executes all three algorithms (GA, SA, TS), compares their fitness scores, and selects the best-performing solution.

3.12 Algorithm Design
3.12.1 Overview of the Hyper-Heuristic Approach
The system employs a selection hyper-heuristic framework, which operates at a higher level of abstraction than individual meta-heuristics. Rather than applying a single optimization algorithm, the system executes three distinct meta-heuristic algorithms on the same scheduling problem and selects the best-performing solution based on fitness evaluation (Burke et al., 2013; Drake et al., 2020).
The three algorithms implemented are:
Genetic Algorithm (GA) — a population-based evolutionary algorithm
Simulated Annealing (SA) — a trajectory-based probabilistic algorithm
Tabu Search (TS) — a memory-based neighbourhood search algorithm
3.12.2 Problem Representation
Each timetable solution is represented as a list of dictionaries, where each dictionary represents a single scheduled session. This representation is consistent across all three algorithms and the fitness function.
timetable = [
    {
        "course": "CSC335",       # Course code
        "lecturer": "Mr. Dada",   # Assigned lecturer
        "students": 176,          # Student count
        "venue": "COMPUTER_LAB",  # Assigned venue
        "room": "COMPUTER_LAB",   # Room identifier
        "capacity": 300,          # Room capacity
        "timeslot": "Wednesday_10-11",  # Day_Hour
        "duration": 1,            # Duration in hours
        "level": "300"            # Student level
    },
    # ... additional sessions
]
The search space is defined by the Cartesian product of available timeslots and venues. With 6 days × 10 hours = 60 timeslots and approximately 30+ venues, each course can potentially be assigned to any of 60 × 30 = 1,800 (day, hour, venue) combinations, making the total search space exponentially large.

3.14 Fitness Function Design
3.14.1 Mathematical Formulation
The fitness function evaluates the quality of a timetable solution by computing a weighted sum of constraint violations. A lower fitness value indicates a better (more feasible) solution. The fitness function is defined as:
$$F(T) = w_1 \cdot V_{LC} + w_2 \cdot V_{RC} + w_3 \cdot V_{CV} + w_4 \cdot V_{PV} + w_5 \cdot V_{US} + w_6 \cdot V_{NLP}$$ Where:
Symbol
Description
Weight ($w_i$)
$V_{LC}$
Number of lecturer clashes
$w_1 = 100,000$
$V_{RC}$
Number of room clashes
$w_2 = 100,000$
$V_{CV}$
Number of capacity violations
$w_3 = 50,000$
$V_{PV}$
Number of practical violations
$w_4 = 10,000$
$V_{US}$
Number of unscheduled sessions
$w_5 = 500,000$
$V_{NLP}$
Number of NLP constraint violations
$w_6 = 200,000$
3.14.2 Violation Detection Logic

The fitness function evaluates timetables by counting three main types of violations. The mathematical formulas below represent how the algorithm penalizes a bad timetable.

1. Lecturer Clash Detection
A lecturer clash occurs when a single lecturer is accidentally scheduled to teach two different courses at the exact same time.

Mathematical Formula:
$$V_{LC} = \sum_{t \in T} \sum_{l \in L} \max(0, count(t, l) - 1)$$
where $count(t, l)$ is the number of courses assigned to lecturer $l$ at timeslot $t$.

Simple Explanation:
- The formula loops through every timeslot and every lecturer.
- For each lecturer in a specific timeslot, it checks how many classes they are teaching.
- If they are teaching 1 class (or 0), the penalty is 0.
- If they are double-booked (e.g., 2 classes at once), it takes 2 - 1 = 1, adding 1 to the total lecturer clash penalty ($V_{LC}$).

2. Room Clash Detection
A room clash happens when two different courses are booked in the exact same classroom at the same time.

Mathematical Formula:
$$V_{RC} = \sum_{t \in T} \sum_{r \in R} \max(0, count(t, r) - 1)$$

Simple Explanation:
- Similar to lecturer clashes, this loops through every timeslot and every room ($r$).
- It counts the number of courses in a specific room at a specific time.
- If there are 2 classes in the same room at the same time, it subtracts 1 to find the number of overlapping classes, adding it to the total room clash penalty ($V_{RC}$).

3. Capacity Violation Detection
A capacity violation occurs when a class has more students enrolled than the classroom can physically seat.

Mathematical Formula:
$$V_{CV} = \sum_{i=1}^{n} \begin{cases} 1 & \text{if } S_i > C_i \\ 0 & \text{otherwise} \end{cases}$$
where $S_i$ is the student count for session $i$ and $C_i$ is the room capacity for session $i$.

Simple Explanation:
- This formula goes through every single scheduled course session ($i$).
- It checks if the number of students ($S_i$) is greater than the room's maximum capacity ($C_i$).
- If there are more students than seats, it adds 1 to the total capacity violation penalty ($V_{CV}$). Otherwise, it adds 0.
3.14.3 Fitness Function Implementation
The following code snippet shows the actual implementation of the fitness function in engine.py:
def evaluate_timetable(timetable, constraints=None):     lecturer_clashes = 0     room_clashes = 0     capacity_violations = 0     practical_violations = 0     unscheduled_sessions = 0     nlp_violations = 0
    lecturer_tracker = {}     room_tracker = {}
    for entry in timetable:         if entry["timeslot"] == "UNSCHEDULED":
            unscheduled_sessions += 1             continue
        lecturer_key = (entry["timeslot"], entry["lecturer"])         room_key = (entry["timeslot"], entry["room"])
        if "UNKNOWN" not in entry["lecturer"].upper():             lecturer_tracker[lecturer_key] = (
                lecturer_tracker.get(lecturer_key, 0) + 1
            )
        room_tracker[room_key] = room_tracker.get(room_key, 0) + 1
        # NLP constraint checking         if constraints:             slot_day = entry["timeslot"].split("_")[0]             if slot_day in constraints.get("avoid_days", []):                 nlp_violations += 1
            lecturer_pref = constraints.get("lecturer_preferences", {})             for lect_name, pref in lecturer_pref.items():
                if lect_name.lower() in entry["lecturer"].lower():                     if pref.get("avoid_day") == slot_day:                         nlp_violations += 1
    for count in lecturer_tracker.values():         if count > 1:             lecturer_clashes += (count - 1)
    for count in room_tracker.values():         if count > 1:             room_clashes += (count - 1)
    for entry in timetable:         if entry["timeslot"] == "UNSCHEDULED":             continue         if entry["students"] > entry["capacity"]:             capacity_violations += 1
    fitness = (
        lecturer_clashes * 100000 +         room_clashes * 100000 +         capacity_violations * 50000 +         practical_violations * 10000 +         unscheduled_sessions * 500000 +
        nlp_violations * 200000     )
    return {
        "fitness": fitness,
        "lecturer_clashes": lecturer_clashes,
        "room_clashes": room_clashes,
        "capacity_violations": capacity_violations,
        "practical_violations": practical_violations,
        "unscheduled_sessions": unscheduled_sessions,
        "nlp_violations": nlp_violations
    }
3.14.4 Weight Justification
The penalty weights are hierarchically assigned based on the severity of each constraint violation:
Unscheduled Sessions ($w_5 = 500,000$): The highest penalty because an unscheduled course represents a complete failure to allocate a required academic activity.
NLP Violations ($w_6 = 200,000$): High penalty to ensure that administrator-specified constraints are respected.
Lecturer and Room Clashes ($w_1 = w_2 = 100,000$): Equal high penalties because both types of clashes render the timetable physically infeasible.
Capacity Violations ($w_3 = 50,000$): Moderate penalty because while overcrowding is undesirable, it does not prevent the class from occurring.
Practical Violations ($w_4 = 10,000$): Lowest penalty for violations related to practical/laboratory session requirements.
3.15 Initial Timetable Generation
3.15.1 Algorithm Description
The initial timetable is generated using a greedy constructive heuristic that processes courses in descending order of student count (largest classes first). This priority-based approach ensures that larger classes are assigned to suitable venues before smaller classes, reducing the likelihood of capacity violations.

3.15.3 Room Selection Strategy
The get_valid_rooms() function implements an intelligent room selection strategy:
Computer Course Detection: Courses with prefixes CSC, CIT, or CIS are identified as computer courses and preferentially assigned to the COMPUTER_LAB venue.
Capacity Filtering: Rooms with capacity less than the student count are filtered out.
Relaxed Capacity: If no room meets the exact capacity requirement, rooms with capacity ≥ 70% of the student count are considered.
Fallback: If no suitable room is found, all venues are made available as candidates.

3.16 Genetic Algorithm Methodology
3.16.1 Algorithm Overview
The Genetic Algorithm (GA) is a population-based evolutionary optimization algorithm inspired by the Darwinian principles of natural selection and genetic inheritance (Katoch et al., 2021). In the context of timetable scheduling, each individual in the population represents a complete timetable solution, and the algorithm iteratively evolves the population toward better solutions through selection, crossover, and mutation operators.

3.16.3 Genetic Operators
Tournament Selection:
Tournament selection works by randomly sampling a subset of the population (tournament size = 5) and selecting the individual with the lowest (best) fitness value. This method provides good selection pressure while maintaining diversity.
def tournament_selection(scored_population, tournament_size=5):
    participants = random.sample(scored_population, tournament_size)     participants.sort(key=lambda x: x[0])     return participants[0][1]
Single-Point Crossover:
The crossover operator creates a child solution by combining genetic material from two parent solutions. A random crossover point is selected, and the child inherits sessions from parent 1 before the point and from parent 2 after the point.
def crossover(parent1, parent2):     size = len(parent1)
    point = random.randint(1, size - 2)
    child = []     for i in range(size):         if i < point:             child.append(copy.deepcopy(parent1[i]))         else:             child.append(copy.deepcopy(parent2[i]))
    return child
Mutation:
The mutation operator modifies individual sessions within a timetable by reassigning their timeslot and venue. Each session has a 20% probability of being mutated. The mutation includes a recovery mechanism that attempts to reschedule any sessions that become unscheduled during mutation.
3.16.4 GA Pseudocode
ALGORITHM: GeneticAlgorithm(df, constraints, pop_size, generations, mutation_rate)
INPUT: Dataset df, constraints, population size, max generations, mutation rate OUTPUT: Best timetable solution, best fitness, fitness history
population ← GENERATE pop_size initial solutions from df
best_solution ← NULL
best_fitness ← +∞
history ← empty list
FOR generation = 0 TO generations - 1:
scored_population ← empty list
FOR EACH individual IN population:
fitness ← EvaluateTimetable(individual, constraints)
APPEND (fitness, individual) TO scored_population
SORT scored_population BY fitness ASCENDING
IF scored_population[0].fitness < best_fitness:
best_fitness ← scored_population[0].fitness
best_solution ← DEEP_COPY(scored_population[0].individual)
APPEND best_fitness TO history
PRINT "Generation {generation} | Best Fitness: {best_fitness}"
// ELITISM: Preserve top 5 individuals
new_population ← TOP 5 from scored_population
// OFFSPRING GENERATION
WHILE |new_population| < pop_size:
parent1 ← TournamentSelection(scored_population, 5)
parent2 ← TournamentSelection(scored_population, 5)
child ← Crossover(parent1, parent2)
child ← Mutate(child, constraints, mutation_rate)
APPEND child TO new_population
population ← new_population
RETURN best_solution, best_fitness, history

3.17 Simulated Annealing Methodology
3.17.1 Algorithm Overview
Simulated Annealing (SA) is a probabilistic meta-heuristic optimization algorithm inspired by the annealing process in metallurgy, where materials are heated and then slowly cooled to reduce defects (Delahaye et al., 2019). In the context of timetable scheduling, SA explores the solution space by iteratively generating neighbouring solutions through mutation and accepting or rejecting them based on the Metropolis acceptance criterion.

3.17.3 Metropolis Acceptance Criterion
The core of SA is the acceptance probability function, which determines whether a neighbouring solution is accepted:
$$P(\text{accept}) = \begin{cases} 1 & \text{if } \Delta E < 0 \text{ (improvement)} \ e^{-\Delta E / T} & \text{if } \Delta E \geq 0 \text{ (deterioration)} \end{cases}$$ Where:
$\Delta E = f(\text{neighbour}) - f(\text{current})$ is the change in fitness
$T$ is the current temperature
At high temperatures, the algorithm accepts deteriorating moves with high probability (encouraging exploration). As the temperature decreases, the acceptance probability for worse solutions approaches zero (encouraging exploitation).
3.17.4 Cooling Schedule
The system uses a geometric cooling schedule:
$$T_{k+1} = \alpha \cdot T_k$$
where $\alpha = 0.995$ is the cooling rate. This ensures a gradual transition from exploration to exploitation over the course of the optimization.

3.18 Tabu Search Methodology
3.18.1 Algorithm Overview
Tabu Search (TS) is a meta-heuristic optimization algorithm that uses an adaptive memory structure (the tabu list) to guide the search away from previously visited solutions, thereby avoiding cycling and encouraging exploration of new regions of the search space (Sörensen et al., 2018). In each iteration, TS generates multiple neighbourhood solutions, selects the best non-tabu candidate, and moves to that solution.

3.18.3 Tabu List Structure
The tabu list stores solution signatures — compact representations of visited timetable configurations. Each signature is a tuple of (course, timeslot) pairs for every session in the timetable:
def timetable_signature(timetable):     return tuple(
        (entry["course"], entry["timeslot"])
        for entry in timetable
    )

3.20 Mutation Operator Design
3.20.1 Mutation Strategy
The mutation operator is a critical component shared by all three algorithms. It generates neighbouring solutions by randomly modifying the timeslot and venue assignments of individual sessions. The mutation includes a two-phase approach:
Phase 1 — Random Reassignment:
Each session has a configurable probability (mutation rate) of being selected for reassignment.
Selected sessions are removed from their current timeslot/room and assigned to a new valid timeslot/room combination that does not conflict with existing assignments.
Phase 2 — Recovery Scheduling:
Any sessions that become unscheduled during Phase 1 undergo a recovery attempt, where the algorithm tries all available timeslot/room combinations to find a feasible assignment.

3.22 Hyper-Heuristic Selection Strategy
3.22.1 Algorithm Selection Mechanism
After all three algorithms complete their optimization runs, the system employs a simple minimum fitness selection strategy to choose the best algorithm:
best_solution = min(
    [
        ("Genetic Algorithm", ga_solution, ga_fitness),
        ("Simulated Annealing", sa_solution, sa_fitness),
        ("Tabu Search", tabu_solution, tabu_fitness)
    ],
    key=lambda x: x[2] )
The algorithm with the lowest fitness score (i.e., fewest weighted constraint violations) is selected as the winning algorithm, and its solution is adopted as the final timetable.

4.7 Performance Analysis
4.7.1 Why the Genetic Algorithm Performed Best
The Genetic Algorithm emerged as the best-performing algorithm for several reasons:
Population Diversity: Unlike SA and TS which maintain a single current solution, GA maintains a population of 50 diverse solutions. This population-level diversity enables more thorough exploration of the search space, increasing the probability of finding high-quality solutions.
Crossover Operator: The single-point crossover combines beneficial scheduling patterns from two different parent solutions, potentially creating children that inherit good timeslot/room assignments from both parents. This recombination capability is absent in SA and TS.
Elitism: The preservation of the top 5 individuals across generations ensures that the bestfound solution is never lost, providing a monotonically non-increasing best fitness trajectory.
Tournament Selection Pressure: Tournament selection with a size of 5 provides sufficient selection pressure to drive the population toward better solutions while maintaining enough diversity to avoid premature convergence.
Balanced Mutation Rate: The 20% mutation rate strikes an effective balance between exploration (modifying existing solutions) and exploitation (preserving good assignments).
4.7.2 Why Simulated Annealing Underperformed
Simulated Annealing's relative underperformance can be attributed to:
Single-Solution Trajectory: SA maintains only one current solution, limiting its ability to explore distant regions of the search space simultaneously.
Acceptance of Deteriorating Moves: While the Metropolis criterion enables escape from local optima, it can also lead the search away from promising regions, particularly at high temperatures.
Parameter Sensitivity: The algorithm's performance is sensitive to the initial temperature and cooling rate. The chosen parameters ($T_0 = 10,000$, $\alpha = 0.995$) may not be optimal for this specific problem instance.
No Recombination: Unlike GA, SA has no mechanism for combining good components from different solutions.
4.7.3 Tabu Search Performance Assessment
Tabu Search performed better than SA but slightly worse than GA:
Effective Cycling Prevention: The tabu list successfully prevented the search from revisiting previously explored solutions, ensuring continuous exploration.
Neighbourhood Diversity: The higher mutation rate (0.3) and multiple neighbour generation (20 per iteration) provided good exploration diversity.
Memory Overhead: The tabu list's signature comparison introduces computational overhead, contributing to the longer execution time.
Limited by Neighbourhood Size: With only 20 neighbours per iteration, the algorithm's exploration is constrained compared to GA's population of 50 individuals.

4.11 Discussion
4.11.1 Effectiveness of the Hyper-Heuristic Approach
The hyper-heuristic framework proved effective in this study by allowing multiple algorithms to compete on the same problem instance and automatically selecting the best-performing solution. This approach addresses a fundamental limitation of single-algorithm systems — namely, the uncertainty about which algorithm will perform best for a given problem instance (Burke et al., 2013). By running all three algorithms and comparing their results, the system guarantees that the administrator receives the best available solution.
4.11.2 Impact of NLP Integration
The integration of Natural Language Processing for constraint specification significantly improved the system's usability compared to traditional form-based or code-based constraint definition. The regex-based NLP parser successfully translated common scheduling constraints from English text to machine-readable formats. However, the current implementation is limited to a predefined vocabulary of constraint patterns and does not support free-form natural language understanding.
4.11.3 Scalability Considerations
The system demonstrated adequate performance for the Covenant University dataset (490+ courses, 
200+ lecturers, 50+ venues). However, for larger institutions with thousands of courses, the computational cost of the Genetic Algorithm (which evaluates 50 individuals per generation, each requiring a full fitness evaluation) may become a bottleneck. Potential optimizations include parallel fitness evaluation, incremental fitness updates, and adaptive population sizing.
4.11.4 Comparison with Related Work
The results obtained in this study align with findings in the literature:
Burke et al. (2013) demonstrated that hyper-heuristic approaches outperform singlealgorithm solutions for timetabling problems, a finding consistent with our comparative results.
Ceschia et al. (2023) established the NP-hard nature of timetable scheduling, justifying the use of meta-heuristic approaches rather than exact algorithms.
Delahaye et al. (2019) demonstrated SA's effectiveness for combinatorial optimization, though our results show GA outperforming SA for this specific problem instance.
Sörensen et al. (2018) documented Tabu Search as an effective memory-based metaheuristic, which our results confirm through TS's superior performance over SA.
4.12 Summary
This chapter presented the implementation details, user interface design, experimental setup, and comprehensive results of the hyper-heuristic timetable generation system. The Genetic Algorithm achieved the best overall performance with a fitness score of 150,000 (94.12% improvement over the initial solution), followed by Tabu Search (250,000, 90.20% improvement) and Simulated Annealing (350,000, 86.27% improvement). All three algorithms successfully scheduled all courses without any unscheduled sessions and eliminated or significantly reduced lecturer and room clashes. The NLP constraint parser correctly extracted all tested constraint patterns, demonstrating the feasibility of integrating natural language reasoning with hyper-heuristic optimization for university timetable generation.
CHAPTER FIVE
SUMMARY, CONCLUSION, AND RECOMMENDATIONS
5.1 Introduction
This chapter presents a comprehensive summary of the research undertaken, the conclusions drawn from the findings, and recommendations for future work. The chapter synthesizes the key contributions of the study titled "From Prompt to Timetable: Integrating Natural Language Reasoning with Different Hyper-Heuristic Algorithms" and reflects on the extent to which the research objectives have been achieved.
5.2 Summary
5.2.1 Problem Statement
The university timetable scheduling problem (UTSP) is a well-documented NP-hard combinatorial optimization problem that requires assigning courses to timeslots and venues while satisfying multiple hard and soft constraints. Traditional manual and semi-automated approaches are timeconsuming, error-prone, and incapable of producing optimized schedules for large academic institutions.
5.2.2 Research Objectives Revisited
This study set out to:
Design and implement an intelligent timetable generation system that integrates natural language processing with hyper-heuristic optimization algorithms.
Develop a Natural Language Processing (NLP) module that enables administrators to specify scheduling constraints in plain English.
Implement three meta-heuristic optimization algorithms — Genetic Algorithm, Simulated Annealing, and Tabu Search — as competing heuristics within a hyper-heuristic framework.
Evaluate and compare the performance of the three algorithms using standardized metrics.
Develop a user-friendly web-based interface for timetable generation, visualization, and management.
5.2.3 Methodology Summary
The study adopted the Design Science Research (DSR) methodology with an experimental approach. The system was developed using Python 3.10+ with Streamlit (frontend), FastAPI (backend), and SQLite (database). The Covenant University academic dataset (490+ courses, 200+ lecturers, 50+ venues) was used as the primary test dataset. Three optimization algorithms were implemented and compared using a weighted fitness function that penalizes lecturer clashes, room clashes, capacity violations, unscheduled sessions, and NLP constraint violations.

5.3.1 Objective 1: Intelligent Timetable Generation System
Status: Achieved ✅
The system successfully generates optimized, clash-minimized university timetables from raw academic datasets. The greedy constructive heuristic produces an initial feasible solution, which is then refined by three competing optimization algorithms. The hyper-heuristic framework automatically selects the best-performing algorithm, ensuring optimal output regardless of which algorithm performs best for a given problem instance.
5.3.2 Objective 2: NLP Constraint Specification
Status: Achieved ✅
The NLP module successfully translates natural language constraint specifications into structured machine-readable JSON objects. The regex-based parser supports lecturer overlap prevention, morning preference, evening avoidance, large hall requirements, day-specific restrictions, and individual lecturer scheduling preferences. All tested constraint patterns were correctly parsed with 100% accuracy.

Status: Achieved ✅
All three meta-heuristic algorithms were successfully implemented as modular, interchangeable components:
Genetic Algorithm: Population-based search with tournament selection, single-point crossover, elitism, and mutation. Achieved the best fitness score (150,000).

5.4 Findings
The major findings from this study are:
Finding 1: Genetic Algorithm Superiority for UTSP
The Genetic Algorithm consistently outperformed Simulated Annealing and Tabu Search in terms of fitness score optimization. This superiority is attributed to:
Population-level diversity enabling broader search space exploration
Crossover operator combining beneficial scheduling patterns from multiple solutions
Elitism preserving the best solutions across generations
Finding 2: All Algorithms Achieve Significant Improvement
All three algorithms achieved substantial improvements over the initial greedy heuristic solution (86 –94% fitness reduction), confirming the effectiveness of meta-heuristic optimization for the university timetable scheduling problem.
Finding 3: Trade-off Between Solution Quality and Execution Time
A clear trade-off exists between solution quality and computational cost:
GA: Best quality (150,000) but moderate execution time (~45s)
SA: Fastest execution (~30s) but lowest quality (350,000)
TS: Longest execution (~55s) with intermediate quality (250,000)
Finding 4: NLP Integration Enhances Usability
The NLP constraint parser significantly simplifies the constraint specification process compared to traditional form-based or code-based approaches. Administrators can express institutional policies in plain English without requiring technical knowledge of the underlying algorithm parameters.
Finding 5: Capacity Violations Are the Most Persistent
While lecturer and room clashes were effectively eliminated by GA and TS, capacity violations persisted across all algorithms (3 violations each). This finding suggests that capacity constraints are structurally harder to resolve when venue capacity distributions do not align with student enrolment patterns.
5.5 Contributions
This study makes the following contributions to knowledge:
5.5.1 Theoretical Contributions
Integration of NLP with Hyper-Heuristics: This study demonstrates a novel approach to combining natural language reasoning with hyper-heuristic optimization for timetable scheduling, extending the traditional hyper-heuristic framework with a natural language constraint specification layer.
Comparative Meta-Heuristic Analysis: The study provides a systematic comparison of GA, SA, and TS for the Nigerian university timetabling context, contributing empirical evidence to the meta-heuristic selection literature.
5.5.2 Practical Contributions
Functional Prototype: A fully operational timetable generation system with NLP integration, multi-algorithm optimization, and a web-based interface.
Dataset: A cleaned and structured academic dataset from Covenant University that can serve as a benchmark for future timetabling research.
Modular Architecture: A modular codebase that allows easy addition of new optimization algorithms, constraint types, and interface features.
5.6 Limitations
Despite the achievements of this study, the following limitations are acknowledged:
NLP Scope: The NLP constraint parser is based on rule-based regex pattern matching and is limited to a predefined vocabulary. It does not support complex or ambiguous natural language inputs. Advanced NLP techniques such as transformer-based language models (e.g., BERT, GPT) were not employed.
Single-Semester Scope: The system generates timetables for a single semester and does not account for multi-semester planning or year-long scheduling considerations.
No Multi-Objective Optimization: The fitness function combines all objectives into a single weighted sum. True multi-objective optimization (e.g., Pareto-based approaches) was not implemented.
Static Dataset: The system operates on a static CSV dataset and does not support real-time data updates or integration with a live student information system.
Limited Constraint Types: The current implementation does not support all possible constraint types (e.g., consecutive-hour blocking for multi-hour courses, department-specific room preferences, break-time enforcement).
Parameter Tuning: The algorithm parameters (population size, mutation rate, cooling rate, tabu list size) were set based on common literature values and preliminary testing, not through systematic hyperparameter optimization.
Single-Institution Testing: The system was tested only on the Covenant University dataset and may require adaptation for institutions with significantly different scheduling structures.
Fragmented and Partially Estimated Dataset: A significant data collection limitation of this study is that the academic dataset used for system testing was not sourced from a single integrated university information system. Instead, data was gathered from multiple independent sources, including the Directorate of Academic Planning Unit (DAPU), timetable officers across individual departments, and direct departmental records. This multi-source collection process introduced several data quality challenges:
(i) Inconsistent or incomplete lecturer assignment records across departments.
(ii) Missing or ambiguous venue capacity information, which in several instances required estimation based on physical inspection and informed judgement rather than official records.
(iii) Lack of standardized data formats across departments, requiring extensive manual cleaning and normalisation before the dataset could be used as system input.
As a result, the generated timetable, while algorithmically optimal given the available data, may not perfectly reflect true institutional scheduling constraints. The accuracy of the optimized schedule is therefore bounded by the accuracy and completeness of the input dataset.
5.7 Conclusion
This study successfully designed, implemented, and evaluated a hyper-heuristic university timetable generation system that integrates Natural Language Processing with three meta-heuristic optimization algorithms — Genetic Algorithm, Simulated Annealing, and Tabu Search. The system demonstrates that:
Natural language reasoning can be effectively integrated with meta-heuristic optimization to create user-friendly scheduling systems that bridge the gap between nontechnical administrators and complex algorithmic optimization.
The hyper-heuristic approach of running multiple algorithms and selecting the bestperforming solution is a robust strategy for timetable scheduling, as the relative performance of algorithms may vary across problem instances.
The Genetic Algorithm is the most effective meta-heuristic for the university timetabling problem in this context, achieving a 94.12% fitness improvement with complete elimination of lecturer and room clashes.
All three meta-heuristic algorithms significantly outperform the initial greedy heuristic, confirming the value of optimization-based approaches for NP-hard scheduling problems.
The modular system architecture supports easy extension with additional algorithms, constraint types, and interface features, making it a flexible platform for future timetabling research.
The system represents a significant step toward automated, intelligent university timetable generation that respects both hard institutional constraints and soft stakeholder preferences expressed in natural language.
5.8 Recommendations
Based on the findings and limitations of this study, the following recommendations are made:
5.8.1 For Institutional Adoption
Pilot Deployment: It is recommended that the system be piloted at the departmental level before full institutional deployment, allowing administrators to familiarize themselves with the interface and validate the generated timetables against known scheduling requirements.
Data Quality Assurance: Institutions should establish standardized course registration data formats and invest in data quality processes to ensure that the input dataset accurately reflects current course offerings, lecturer assignments, and venue capacities.
Training: Academic administrators should receive brief training on the system's NLP constraint specification capabilities to maximize the effectiveness of the natural language interface.
5.8.2 For Future Development
Advanced NLP Integration: Future versions should incorporate transformer-based language models (e.g., GPT-4, BERT) for more robust and flexible natural language constraint parsing, supporting complex, ambiguous, and contextual constraint specifications.
Multi-Objective Optimization: Implement Pareto-based multi-objective optimization to simultaneously optimize multiple competing objectives without requiring predefined weight assignments.
Real-Time Data Integration: Connect the system to institutional student information systems (SIS) and learning management systems (LMS) for real-time data access and automatic schedule updates.
Cloud Deployment: Deploy the system on cloud infrastructure (e.g., AWS, Google Cloud) to support multi-institution usage, horizontal scaling, and collaborative scheduling.
Additional Algorithms: Extend the hyper-heuristic framework with additional metaheuristics such as Ant Colony Optimization (ACO), Particle Swarm Optimization (PSO), and Harmony Search.
5.9 Future Work
The following areas represent promising directions for future research:
Deep Learning-Based Constraint Learning: Develop machine learning models that automatically learn institutional scheduling patterns from historical timetables, reducing the need for explicit constraint specification.
Hybrid Algorithm Design: Investigate hybrid approaches that combine the strengths of multiple algorithms (e.g., GA-SA hybrid, GA-TS hybrid) within a single optimization framework.
Dynamic Rescheduling: Develop algorithms for dynamic timetable adjustment in response to real-time events such as lecturer absences, room closures, or emergency schedule changes.
Exam Timetabling Extension: Extend the system to support examination scheduling, which introduces additional constraints such as student exam spacing, invigilator assignment, and conflict-free exam periods.
Mobile Application: Develop a companion mobile application for lecturers and students to view their personalized schedules, receive notifications about schedule changes, and provide feedback.
Reinforcement Learning: Explore the application of reinforcement learning for adaptive algorithm selection in the hyper-heuristic framework, where the system learns which algorithm performs best for different types of scheduling problems over time.
Constraint Satisfaction Problem (CSP) Integration: Integrate CSP solvers as an additional layer of constraint verification before meta-heuristic optimization, ensuring hard constraint feasibility at every stage.
Institutional Data Integration and Synchronisation: Future work should prioritise the development of a centralised institutional data pipeline that draws directly from a unified University Management Information System (UMIS). Rather than relying on data collected from fragmented sources such as individual departmental timetable officers and the Directorate of Academic Planning Unit (DAPU), future implementations should:
(i) Integrate with the university's official student registration system to obtain verified course enrolment figures.
(ii) Connect to a verified venue capacity database maintained by the university's facilities management unit, eliminating the need for estimated capacity values.
(iii) Establish an automated data synchronisation mechanism that updates course, lecturer, and venue records in real time, ensuring that the timetable generation system always operates on the most current and accurate institutional data.
Such an integrated data foundation would substantially improve the accuracy, fairness, and institutional validity of the generated timetables, and would represent a critical step toward full production-readiness for institutions of any scale.
